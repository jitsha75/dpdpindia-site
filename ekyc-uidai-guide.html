01

eKYC Ecosystem Overview

Entity hierarchy from UIDAI/CIDR down to the resident

Key rule: A Sub-KUA has no direct agreement with UIDAI. Its KUA is contractually responsible for it. eKYC data received by a KUA may be shared with its Sub-KUAs but cannot be transferred beyond — Rule 16 strictly prohibits further onward sharing.
02

Entity Roles & Responsibilities

What each participant is permitted to do, and what they cannot

UIDAI & CIDR

The root regulator and data custodian. Issues all licences, maintains the Central Identities Data Repository, and is the only entity that ever performs biometric matching or decrypts the PID block.

  • Licences ASAs, KUAs, and approves Sub-KUAs
  • Operates CIDR — stores encrypted biometric templates
  • Returns signed, encrypted eKYC XML to requesting KUA
  • Provides masked Aadhaar (last 4 digits) in response
Root CA Biometric vault Regulator

ASA / KSA (Aadhaar Service Agency / KYC Service Agency)

The physical connectivity layer. Maintains dedicated, registered leased-line connections between KUAs and CIDR. Cannot decrypt any payload — it is a pure routing entity.

  • Must register its static IP(s) with UIDAI
  • Hosts dedicated server solely for Aadhaar authentication
  • Manages cryptographic key distribution for KUAs
  • Cannot read the PID block or KYC response
  • Is labelled KSA specifically in the eKYC context
Leased-line only Cannot decrypt Key distribution

KUA (KYC User Agency)

A licensed entity (bank, NBFC, telecom, insurer) that directly consumes the eKYC service. Holds a private key in an HSM to decrypt the UIDAI-signed eKYC response.

  • Direct contractual relationship with UIDAI
  • Assigns licence code ac used in every eKYC request
  • Must store Aadhaar number in ADV only
  • Must NOT log raw KYC fields or Aadhaar number
  • Responsible for all its Sub-KUAs
  • Private key held in FIPS 140-2 certified HSM
HSM private key ADV required Regulation 16

Sub-KUA

An entity that uses the KUA's infrastructure to perform eKYC. No direct UIDAI contract. KUA must get UIDAI prior approval before onboarding any Sub-KUA. Examples: insurers using a bank's eKYC, brokers on an exchange's platform.

  • Identified in requests via sa attribute
  • Cannot receive eKYC data beyond what KUA passes
  • Cannot onward-share eKYC data to any further party
  • Subject to same data-protection obligations as KUA
sa attribute No direct UIDAI contract No onward sharing

AUA / Sub-AUA (Auth User Agency)

Authentication-only variant. Gets a binary yes/no on identity — no demographic data. A KUA is always an AUA, but an AUA is not necessarily a KUA (no eKYC entitlement).

  • Authentication response: Y/N only, no name/address/photo
  • Uses Auth API, not eKYC API
  • Sub-AUAs use an AUA's infrastructure identically to Sub-KUA model
Y/N only Auth API No KYC data

Resident (Aadhaar Holder)

The data principal. The eKYC transaction is valid only with explicit, informed, purpose-specific consent. The resident controls what data is shared and for which purpose.

  • Must explicitly consent: rc='Y' in request
  • Biometric captured only on STQC-certified Registered Device
  • OTP delivered only to Aadhaar-registered mobile
  • Cannot share eKYC data with anyone beyond the KUA/Sub-KUA scope
rc='Y' consent Data principal STQC device

Bank Application Log Requirements

What a bank (KUA) log must contain — and what is legally prohibited

Per UIDAI eKYC API specification v2.5 and Aadhaar Authentication Regulations 2021, the following governs what a KUA's application logs may and may not record:

✓ Required / Permitted in logs

✓txn — Transaction ID
✓ts — Request timestamp (ISO 8601)
✓ac — KUA licence code
✓sa — Sub-KUA code (if applicable)
✓auth-type: BIO / OTP / DEMO flag
✓uid-token — UIDAI-issued pseudonym, not Aadhaar no.
✓resp-code — UIDAI response/error code
✓err — Error code if failed
✓device-id — Registered Device ID
✓stqc-cert-ref — Device certification ref
✓rc='Y' — Consent captured flag
✓ASA identifier used for routing
✓Customer internal reference number
✓ttl — Time-to-live of response

✗ Strictly Prohibited in logs

✗Aadhaar number (12 digits)
✗Biometric data — fingerprint / iris template
✗OTP value
✗Decrypted PID block content
✗Full eKYC XML in application DB
✗Name from eKYC response
✗Date of birth from eKYC response
✗Address details from eKYC response
✗Photo (JPEG) from eKYC response
✗Mobile number (even hashed form in logs)
✗Email address from eKYC response
✗Session AES encryption keys
✗wadh (anti-replay token value)
Aadhaar Data Vault (ADV) rule: The Aadhaar number itself may only be stored in a separately hardened Aadhaar Data Vault. The bank's operational application databases and logs must never contain Aadhaar numbers — only the UIDAI-issued uid-token (a pseudonymous reference unique per KUA) may appear. The eKYC XML signed by UIDAI is stored as-is in the ADV, encrypted with AES-256 and access-controlled via HSM.
Practical implication: A bank's fraud team or compliance officer auditing the transaction log can see that customer reference 12345 completed eKYC at 14:32 IST via biometric, the UIDAI returned success (uid-token XYZ), on device D123 — but cannot retrieve the customer's Aadhaar number, name, or address from those logs. That data exists only in the ADV and the separately secured KYC record store.

Data visibility at every entity in the chain

Sub-KUA server Encrypted in PID Encrypted in PID Receives from KUA ¹ In ADV only

UIDAI CIDR Full access (1:1 match) Decrypts → matches only² Generates + encrypts to KUA Signs with UIDAI private key
¹ Sub-KUA receives demographic data forwarded by KUA after KUA decrypts — KUA acts as the trust boundary. Sub-KUA may NOT receive biometric data under any circumstance.
² CIDR decrypts biometric only for the duration of the matching computation. The biometric is never exported, logged, or stored in decrypted form outside CIDR.
Design principle — layered encryption: The PID block is encrypted with UIDAI's public key (RSA-2048) at the point of capture. This means no intermediary — not the KUA, not the ASA — can decrypt or read the biometric even if they intercept the packet. The eKYC response from UIDAI is encrypted with the KUA's public key, so the ASA cannot read the KYC data in transit. Two independent encryption layers protect two different categories of sensitive data, and each is addressed only to the entity that legitimately needs it.
06

How Biometric & Aadhaar Leakage Is Prevented

Technical and architectural controls at each point in the chain

1. Registered Devices (RD) — L1 / L2 specification

Every device used to capture biometrics for Aadhaar authentication must be a STQC-certified Registered Device. L1 devices (higher security tier) perform all encryption operations inside a Trusted Execution Environment (TEE) — the captured biometric never exists in a readable form outside that secure chip. The device embeds a device public certificate, and the wadh anti-replay token is constructed from the device key, session key, and Aadhaar number hash — meaning a replayed packet from a previous session is cryptographically invalid.

2. PID Block — the biometric envelope

The Personal Identity Data (PID) block is the container that holds biometric data or OTP before transmission. Construction:

  • A fresh AES-256 session key is generated per transaction.
  • The biometric (or OTP) is encrypted with this session key.
  • The session key is encrypted with UIDAI's RSA-2048 public key.
  • The combined encrypted PID block (Data + Hmac + SessionKey) is embedded in the Auth XML.
  • Only UIDAI's CIDR, which holds the corresponding private key, can open it.

3. Encryption standards

UIDAI mandates: RSA-2048 for asymmetric (public key) operations, AES-256-GCM for symmetric encryption. All connections use TLS. UIDAI rotates its public key periodically; KUAs must update to the current certificate before it expires.

4. HMAC verification

The Hmac field in the PID block contains a SHA-256 HMAC of the PID data using the session key. CIDR verifies this HMAC before attempting decryption, providing integrity assurance — a tampered PID block is rejected before any biometric comparison.

5. wadh — anti-replay token

The wadh (Witness Authentication Data Hash) is computed as SHA-256(deviceKey + sessionKey + Aadhaar_hash). CIDR rejects any request where the wadh does not match, and CIDR maintains a nonce store to detect replayed transactions within the TTL window.

6. KUA digital signature on Auth XML

The entire Auth/eKYC XML is digitally signed by the KUA's private key (stored in an HSM). CIDR verifies this signature before processing, ensuring the request originates from the licenced KUA and has not been tampered with in transit via the ASA.

7. UIDAI digital signature on eKYC response

The KYC response XML is signed with UIDAI's private key using W3C XML-DSig. The KUA must verify this signature against UIDAI's published public certificate before accepting the response. This allows any downstream party (court, regulator) to independently verify that the eKYC data was issued by UIDAI — the signed XML is the legal equivalent of a notarised document.

8. Aadhaar Data Vault (ADV)

KUAs and Sub-KUAs may only store Aadhaar numbers inside an ADV — a separately hardened system from the main application database. The ADV is:

  • Encrypted with AES-256 with keys held in an HSM
  • Access-controlled on a need-to-know basis with audit logging
  • Physically or logically isolated from operational application servers
  • Permitted hosting: RE's own secure premises, MeitY GCC-empanelled cloud, or ADV-as-a-service from UIDAI-approved provider
  • The application replaces Aadhaar numbers with a uid-token (pseudonym) in all other databases and logs

9. Absolute prohibition on biometric storage

Regulation 17(1)(a) — Aadhaar Authentication Regulations 2021 No requesting entity shall store, use, share or publish biometric data collected from a resident for the purposes of Aadhaar authentication under any circumstance. Violation is a punishable offence under Section 40 of the Aadhaar Act, 2016 (imprisonment up to 3 years and/or fine up to ₹10 lakh).

10. Time-to-live (TTL) enforcement

Every eKYC response includes a ttl (time-to-live) value. KUAs must reject any response whose TTL has expired. This prevents stale or cached responses from being replayed in later transactions.

07

Legal & Regulatory Obligations Summary

Key rules from Aadhaar Act 2016 and Authentication Regulations 2021

Section 8 — Consent mandatory A requesting entity can use the Aadhaar authentication service only after obtaining explicit consent from the resident for the stated purpose. Consent must be captured as rc='Y' in the eKYC XML.
Regulation 16 — No onward transfer eKYC data obtained by a KUA may be shared with authorised Sub-KUAs only. It cannot be transferred, sold, or disclosed to any other entity. Prior UIDAI approval is required for each Sub-KUA relationship.
Regulation 17(1)(a) — Biometric prohibition No biometric data may be stored, shared, published, or used by any requesting entity after the authentication event. Biometrics must be discarded immediately after transmission. Violation: up to 3 years imprisonment and/or ₹10 lakh fine.
Regulation 17(1)(b) — Purpose limitation Authentication and eKYC data must be used only for the stated purpose for which resident consent was obtained. Using KYC data obtained for account opening for credit scoring or marketing without separate consent is prohibited.
UIDAI Circular — ADV hosting (2025) KUAs may host their Aadhaar Data Vault only in three permitted environments: their own secure on-premises infrastructure, MeitY GCC-empanelled cloud, or an UIDAI-approved ADV-as-a-service provider. No other hosting is permitted.
Regulation 19 — Audit & reporting Every KUA must maintain a security audit trail of all eKYC transactions (using the fields listed in Section 4 above) and submit periodic compliance reports to UIDAI. Failure to maintain adequate audit trails is grounds for licence suspension.
