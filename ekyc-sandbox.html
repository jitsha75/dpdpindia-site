Sandbox Overview & Purpose

A risk-free Test CIDR environment — no real data, no leased lines, no prior licence required to start

The UIDAI Aadhaar Authentication Sandbox is a dedicated staging environment hosting a mock CIDR (Central Identities Data Repository) populated exclusively with dummy test data. It is designed as the lowest-barrier entry point into the Aadhaar authentication ecosystem — allowing developers, fintechs, and prospective AUAs to validate their integration logic before touching the production environment.

Sandbox / Staging Environment
API versionAuth & eKYC API 2.5 (current)
NetworkPublic internet — no VPN, no leased line required
DataDummy Aadhaar numbers (9999-series) only
BiometricsRD Service Simulation Mode generates valid test PID blocks
CertificatesUIDAI Staging Public Key (.cer)
AccessOpen (public) or registered (post-application)
PurposePoC development, logic validation, pre-production audit prep
Production Environment
NetworkDedicated leased line via registered ASA only
DataReal Aadhaar numbers — all 1.4B residents
BiometricsLive STQC-certified L1 Registered Device required
CertificatesUIDAI Production Public Key
AccessRequires AUA/KUA licence, Joint Agreement, CERT-IN audit pass
PurposeLive identity verification for real users
Critical isolation guarantee: Sandbox data (9999-series UIDs) will never authenticate in production, and real Aadhaar numbers will not work in the sandbox. The two environments are completely isolated — there is no risk of accidentally testing against live resident data.
02

Staging Endpoints

Three API URLs — replace these with production URLs only after CERT-IN audit

The sandbox is not a software download — it is a set of hosted staging URLs maintained by UIDAI at developer.uidai.gov.in. These connect to the Test CIDR backend.

POST Authentication (Auth)
developer.uidai.gov.in
/authserver/2.5

Biometric, OTP, or demographic authentication. Returns Y/N with error code. Used by AUAs.

POST e-KYC (Data Retrieval)
developer.uidai.gov.in
/uidkyc/kyc/2.5

Full demographic data retrieval — Name, DOB, Address, Photo. Used by KUAs. Returns encrypted, signed KycRes XML.

POST OTP Generation
developer.uidai.gov.in
/uidotp/2.5

Triggers an OTP dispatch to the Aadhaar-registered mobile. Call this before submitting an OTP Auth request.

Production switch: When moving to production, you replace these three URLs and the Staging Public Key certificate with the production equivalents provided by UIDAI after your CERT-IN audit passes. No other code change is required if your integration is correctly parameterised.

Access Modes: Who Can Use the Sandbox?

UIDAI operates a dual-access model — open PoC testing and registered pre-production access

Access Type Who It's For Requirement What You Get
Open / Public Any developer, startup, student, or organisation in the Proof of Concept (PoC) stage. No licence, no registered company, no prior permission needed. None. Use the public AUA code and the test license key immediately. Download the SDK and start firing requests. Full access to all three staging endpoints using the public AUA code. Can test Auth, OTP, and eKYC API flows end-to-end with dummy UIDs.
Registered Companies that have formally applied for an AUA or KUA licence and paid the application fee. This is used during the onboarding and pre-production audit phase. Granted by UIDAI during the onboarding process, after initial application submission. Requires a formal AUA/KUA application on file. A specific Sandbox ID tied to your company's AUA code. Allows testing with your own code, preparing for the Self-Certification tests required before production go-live.
Where to start: Every integration begins with Open Access. Use the public AUA code to validate your XML construction, PID block encryption, and API connectivity before investing in the formal licence application. This de-risks the onboarding process significantly.

Technical Prerequisites & Configuration

What your application needs before it can speak to the staging environment

Your application must perform a correct cryptographic handshake with the UIDAI staging server. These are the components required before you can send your first request.

A — Technical Prerequisites
Requirement Detail
Internet connectivity Standard public internet — no VPN or leased line required for the sandbox environment.
Staging Public Key Download the Staging Public Key (.cer file) from the UIDAI Developer Portal. This is used to encrypt the PID block so only the UIDAI staging server can decrypt it. Do not confuse with the Production Public Key.
Digital Signature cert A .p12 or .jks keystore file for signing your XML requests. Sample test certificates are included in the UIDAI tutorial ZIP files — use these for sandbox testing. Production will require your own registered certificate.
Runtime Java 1.6 or higher is recommended — UIDAI provides Java-based SDKs. Maven is recommended for dependency management. .NET clients are also available on the developer portal.
B — Standard Configuration Parameters (Public Access)
Parameter Value Notes
AUA Code (ac) public Use this for all open-access testing. Your own AUA code is issued only after licence approval.
Sub-AUA Code (sa) public Set to match the AUA code for public testing. In production, this is your registered Sub-KUA code.
License Key (lk) MG41K999H9S Common public test key. Always verify the current active key on the Testing Data page — it is updated periodically.
Terminal ID Any string Example: TestTerminal01. No validation is performed on this field in the sandbox.
Transaction ID (txn) Any unique string Must begin with UKC: prefix for eKYC requests. E.g., UKC:TEST:20260319:001.
05

Implementation Steps

Step-by-step walkthrough from environment setup to firing your first authenticated request

A
Environment Setup
Download the SDK, certificates, and configure your build environment
  1. Ensure Java 1.6+ is installed and JAVA_HOME is set. Maven 3.x is recommended for dependency management.
  2. Navigate to the UIDAI Developer Section and download the auth-client-source.zip. This contains the XML construction, PID block encryption, and digital signing logic.
  3. Download the Staging Public Key (.cer file) from the same portal. Import it into your application's truststore: keytool -import -alias uidai-staging -file staging-public.cer -keystore truststore.jks
  4. Download the tutorial ZIP which contains sample .p12 test certificates for signing. Import into your keystore for signing XML requests.
Pro tip: Before writing a single line of custom code, use the UIDAI Sample GUI App (included in the tutorial ZIP) or Postman to fire a test request. This validates that your network connectivity, certificates, and credentials are correct in isolation — eliminating environmental issues before they contaminate your code debugging.
B
Construct the Auth/eKYC XML Request
Build the PID block and Auth XML with correct parameters

Every request to UIDAI is a digitally signed XML document containing an encrypted PID block. The SDK handles this construction — configure it with your parameters:

Configuration — Public Sandbox Access
// AUA and auth settings
auaCode         = "public"
subAuaCode      = "public"
licenseKey      = "MG41K999H9S"
authEndpoint    = "https://developer.uidai.gov.in/authserver/2.5"
eKycEndpoint   = "https://developer.uidai.gov.in/uidkyc/kyc/2.5"
otpEndpoint     = "https://developer.uidai.gov.in/uidotp/2.5"

// Certificates
stagingPublicKey = "staging-public.cer"   // Encrypt PID block
signingKeystore  = "test-signing.p12"     // Sign Auth XML

// Test UID for Demo Auth
testUID         = "999999990019"
txnId           = "UKC:TEST:20260319:001"

The PID block is constructed as follows: the user's demographic data or OTP is placed in a <Pid> element, encrypted with a fresh AES-256 session key, and the session key is wrapped with the UIDAI Staging Public Key. The combined block is Base64-encoded and placed inside the Auth XML.

C
Send the Request & Handle the Response
HTTPS POST to the staging endpoint, parse the Auth or KycRes XML
  1. Digital signing: Sign the complete Auth XML using the test .p12 certificate. The SDK's signXml() method handles this.
  2. HTTPS POST: Send the signed XML as the request body to the appropriate staging endpoint.
  3. Parse response: The response is a signed XML document. Check the ret attribute first — "y" for success, "n" for failure.
  4. Error handling: On failure, parse the err attribute for the error code. See the response table in Section 07.

Test Data Reference

Dummy Aadhaar numbers for sandbox use — real UIDs will not work here

Critical constraint: You cannot test the sandbox with your own real Aadhaar number. The sandbox Test CIDR only contains the 9999-series dummy UIDs. Real Aadhaar numbers will return an error. Download the full test UID list from the Testing Data and License Keys page on the developer portal.
Sample Test UIDs
Dummy UID Name in Test CIDR Suitable for Testing
999999990019 Shivshankar Choudhury Demo Auth, OTP Auth, eKYC
9999-series Multiple test personas Full list in Testing Data PDF on developer portal

Biometric simulation: Since the sandbox does not store real biometric templates, you cannot use a live fingerprint reader. Instead, configure an L0 or L1 Registered Device (RD) Service in Simulation Mode — this generates a valid, correctly-formatted encrypted PID block that the staging CIDR will accept for biometric auth testing.

07

Response Handling & Error Codes

Parsing AuthRes and KycRes XML — common error codes you will encounter

Every response from UIDAI is a digitally signed XML document. Always verify the UIDAI signature before trusting the response content. Check the ret attribute for the outcome, then the err attribute for the specific reason if authentication failed.

ret value err code Meaning & Action
y 0 Authentication or eKYC successful. For eKYC, decrypt the Rar element using your private key to access the KYC payload.
n 100 Name mismatch. The name in the PID block does not match the record in CIDR. Check the demo data you are supplying.
n 300 Biometric mismatch. The fingerprint or iris template did not match. In sandbox, ensure your RD Service is in Simulation Mode and generating valid test PID blocks.
n 400 Invalid OTP. OTP has expired or does not match. Call the OTP endpoint first and use the OTP immediately.
n 500 Invalid AUA/license key. Verify your ac and lk values. Check the Testing Data page for the current public license key.
n 540 Invalid UID. You are using a real Aadhaar number. Switch to a 9999-series test UID.
n 940 Signature verification failed. Your XML signing is incorrect. Check that you are using the correct test .p12 certificate and that the XML is not being modified after signing.
The full error code reference is published in the UIDAI Authentication API specification v2.5, available on the developer portal. The sandbox and production environments use the same error code set.

Security Best Practices in the Sandbox

Habits formed in the sandbox carry directly into production — build compliance in from day one

🔒
No PII in logs — even in sandbox
Even though sandbox UIDs are dummy data, build your logging layer to exclude Aadhaar numbers, biometric data, OTP values, and demographic fields from day one. These habits must carry into production — retrofitting compliant logging is far more expensive than building it correctly initially.
🔑
Always encrypt the PID block
The PID block must always be encrypted using the latest UIDAI Staging Public Key before transmission. Never transmit a plaintext PID block, even for debugging. If you need to inspect what you are sending, decrypt a copy locally — never disable encryption on the outgoing request.
⚠️
Rotate the staging public key
UIDAI periodically updates the Staging Public Key. If your PID block encryption starts failing, check the developer portal for a key update. Build a key-refresh mechanism into your application rather than hard-coding the key — the same will be required in production.
✅
Verify the UIDAI response signature
Always verify the digital signature on every AuthRes and KycRes XML before accepting or processing its contents. This guards against man-in-the-middle attacks and tampered responses. Use the UIDAI Staging Certificate (separate from the Public Key) for response signature verification.
📋
Prepare your CERT-IN audit evidence
The sandbox period is your window to build the log samples, architecture diagrams, and security control documentation that a CERT-IN empanelled auditor will review before you can go to production. Document every security decision made during sandbox development — it becomes your audit evidence.
🚫
Never use real Aadhaar numbers for testing
Using real Aadhaar numbers in the sandbox — even your own — is a violation of the purpose limitation principle and may be viewed as unauthorised processing. The sandbox's 9999-series UIDs exist precisely so that no real resident's identity data is exposed during development. Use only the dummy UIDs.

Quick-Start Checklist

Five steps to get your first successful authentication in under an hour

  • 1
    Visit the UIDAI Developer Section
    Go to uidai.gov.in → Ecosystem → Developer Section. Bookmark this page — it is the central source for all SDKs, certificates, test data, and tutorial guides.
  • 2
    Download the Testing Data PDF
    From the Testing Data and License Keys page, download the PDF listing all dummy Aadhaar numbers. Verify the current public license key on this page — MG41K999H9S is the commonly referenced key but it is updated periodically.
  • 3
    Download the Sample Client Code
    Get the Java Auth Client or .NET Client source code from the Developing using Sample Applications tutorial page. This includes all XML construction, PID encryption, and signing logic pre-built.
  • 4
    Import the Staging Certificates
    Download the Staging Public Key (.cer) and import it into your application truststore. Import the sample test .p12 signing certificate from the tutorial ZIP. Configure the staging endpoint URLs in your application.
  • 5
    Fire a Demo Auth Request
    Use test UID 999999990019 (Shivshankar Choudhury) and perform a Demo Auth — matching just the name field. This is the simplest auth type and has no biometric dependency. A successful ret="y" response confirms your connectivity, certificate setup, and XML construction are all correct.
