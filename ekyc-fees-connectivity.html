01

Licence Fee Structure

UIDAI's slab-based model — 2-year licence validity, fees vary by entity type and transaction volume

UIDAI utilises a slab-based fee model to encourage smaller entities while ensuring high-volume users contribute proportionately to infrastructure costs. Licences are valid for two years and must be renewed before expiry to avoid service interruption.

Entity Type Annual Transaction Slab Licence Fee Validity
KUA / AUA Up to 5 Lakh transactions per year ₹5 Lakh + GST 2 Years
KUA / AUA 5 Lakh to 20 Lakh transactions per year ₹10 Lakh + GST 2 Years
KUA / AUA Above 20 Lakh transactions per year ₹20 Lakh + GST 2 Years
Sub-KUA Standard — commercial entity ₹3 Lakh + GST 2 Years
LITE Code Government entities — fewer than 1 Lakh transactions/year Exempted Annual Review
Important: Licence fee slabs are assessed at the time of application or renewal based on projected or actual transaction volume. Exceeding your slab volume during the licence period may trigger a fee upgrade notice from UIDAI. Monitor monthly transaction counts against your slab ceiling.
Sub-KUA flat fee: The ₹3 Lakh fee for Sub-KUAs is a flat commercial rate regardless of transaction volume — unlike the KUA slab structure. This reflects that the Sub-KUA operates under the KUA's infrastructure and the KUA's own licence covers the connectivity and CIDR interaction costs.

Regulation 25 — Financial Disincentives: UIDAI may levy financial penalties on AUAs/KUAs whose systems generate repeated authentication failures due to technical misconfigurations, incorrect PID block construction, invalid certificate usage, or system errors. This is distinct from genuine resident-side failures (biometric mismatch, wrong OTP). Monitor your failure rate by error code and remediate systemic technical failures promptly.
03

Physical Data Centre Requirements

Infrastructure standards under Compliance Checklist v2.0 — applicable to both Sub-KUA-owned and KUA-shared infrastructure

A Sub-KUA may either build its own dedicated infrastructure or operate within the KUA's shared infrastructure. Either way, the physical hosting environment must meet the following standards — verified by CERT-IN auditors during the mandatory IS Audit.

Option A
Sub-KUA owns its own infrastructure
Sub-KUA builds and maintains its own data centre, ADV, and HSM. Must meet all physical and logical requirements in this article independently. Higher upfront cost but full control.
Option B
Sub-KUA uses KUA's shared infrastructure
Sub-KUA operates within the KUA's existing secure environment. KUA's data centre, HSM logical partition, and network infrastructure are shared under clear contractual boundaries. Lower cost, faster setup.
Physical Security Requirements — Either Option
1
Isolated Zone — dedicated lockable server cabinet
Servers must be kept in a dedicated, lockable cabinet within a secure data centre. The cabinet must not share space with unrelated workloads without logical isolation controls.
2
24/7 Monitoring — security guards and CCTV surveillance
The facility must have round-the-clock physical security guards and CCTV surveillance. CCTV must cover all entry/exit points and all server rack areas with no blind spots.
3
Access Control — biometric or card with timestamped logs
Biometric or card-based access is mandatory with entry/exit logs recording date, time, and stated purpose. Logs must be available for audit review for the preceding 6 months minimum.
04

ADV & HSM Obligations for Sub-KUAs

Critical components required whenever a Sub-KUA stores any Aadhaar data

Trigger condition: If a Sub-KUA stores any Aadhaar-related data — including eKYC XML files, tokenised Aadhaar references, or demographic KYC records — they must implement both an ADV and an HSM. There is no waiver for smaller Sub-KUAs. The only alternative is to not store any Aadhaar data at all and process it statelessly — which is impractical for most real-world use cases.
Aadhaar Data Vault (ADV)

A logically separate database where Aadhaar numbers are replaced by Reference Keys (Tokens). The real Aadhaar numbers must never reside in the business application database — the application stores only the token, and looks up the Aadhaar number in the ADV only when absolutely necessary through an authorised API call.

Business DB: customer_id → token_xyz
ADV: token_xyz → [encrypted Aadhaar]
Hardware Security Module (HSM)

A FIPS 140-2 Level 3 certified device that holds all encryption keys for the ADV and for signing eKYC requests. Private keys must stay inside the HSM — they may never be extracted to a file, environment variable, or application memory.

Cost note: Sub-KUAs are permitted to use a logical partition on the KUA's HSM rather than purchasing their own device — a significant cost saving, provided the KUA agrees and the partition is properly isolated.
05

Connectivity Standards — KUA to Sub-KUA

The link between a KUA's central server and the Sub-KUA's systems must be secured against man-in-the-middle attacks

The connection between the KUA (e.g., a bank's central server) and the Sub-KUA (e.g., a partner application or regional department) must be a secure, encrypted channel. The appropriate connectivity type depends on the volume, permanence, and architecture of the Sub-KUA's deployment.

Connectivity Type Use Case Security Requirement
Leased Line High-volume, permanent sites — e.g., a Sub-KUA with dedicated premises processing large transaction volumes daily Dedicated point-to-point link with no public internet exposure. Equivalent to an ASA leased line. Highest security posture; highest cost.
VPN Small offices, remote applications, or Sub-KUAs with moderate transaction volumes operating from multiple sites Site-to-Site IPsec VPN or SSL VPN with strong encryption (AES-256). Pre-shared keys must be rotated periodically. Certificate-based authentication is preferred over PSK.
HTTPS REST Modern cloud-to-cloud integration — e.g., a Sub-KUA SaaS platform calling the KUA's API gateway Must use TLS 1.3+ and Mutual Authentication (mTLS) where both the KUA and Sub-KUA present certificates. One-way TLS is insufficient. Certificate pinning on the client side is strongly recommended.
Critical regardless of connectivity type: Even when using a secure VPN or leased line, the PID block (biometric or OTP data) must remain encrypted with UIDAI's public key throughout the entire journey from the Sub-KUA to the KUA to CIDR. The secure channel is not a substitute for PID block encryption — both are required simultaneously.
06

Network Segregation Rules

IP whitelisting, DMZ placement, and PID block encryption — three non-negotiable requirements

1
IP Whitelisting — KUA server accepts only Sub-KUA's static IPs
The KUA server must be configured to only accept traffic from the specific static IP addresses registered for the Sub-KUA. Dynamic IPs, IP ranges, or geographic-based allowlists are not acceptable under Compliance Checklist v2.0. Any traffic from an unlisted IP must be silently dropped at the firewall — not rejected with an error that could leak infrastructure information.
2
Firewall & DMZ — Sub-KUA Aadhaar application in a demilitarised zone
The Sub-KUA application that processes Aadhaar transactions must reside in a DMZ (Demilitarised Zone) separated from the general corporate network by a dedicated firewall. The DMZ must allow only the specific ports and protocols required for eKYC — all other traffic must be denied by default. Firewall rules must be documented and approved by the CISO.
3
Encrypted PID Block — encrypted throughout, regardless of channel security
Even with a secure VPN or leased line, the PID block (biometric or OTP data) must remain encrypted with UIDAI's public key at all times — from the point of capture on the Registered Device through the Sub-KUA system, across the KUA connection, and all the way to CIDR. The channel encryption and the PID encryption are independent layers, and both are mandatory simultaneously. Removing PID encryption "because the channel is secure" is a compliance violation.
07

Onboarding Process — 2026 Update

The four-step process to appoint a Sub-KUA under the current regulatory framework

1
Joint Undertaking — legally binding shared liability agreement
Both the KUA (e.g., the bank) and the proposed Sub-KUA must sign a Joint Undertaking accepting joint and several liability for any data security breach, misuse, or regulatory violation arising from the Sub-KUA's operation. This document is not a standard NDA — it is a specific UIDAI-format agreement. Obtain the current template from the UIDAI ecosystem portal. Both organisations' authorised signatories must sign.
Prerequisite for all subsequent steps
2
UIDAI Application — formal "Sub-KUA Appointment Form" submission
The KUA submits the Sub-KUA Appointment Form to UIDAI, accompanied by the signed Joint Undertaking, the Sub-KUA's business justification for needing eKYC access, and the proposed data architecture. The application must demonstrate a legitimate purpose — UIDAI does not approve Sub-KUA appointments for entities whose business case does not require eKYC data. The applicable licence fee must be submitted with the application.
3
CERT-IN Audit — infrastructure verification before any live transaction
The Sub-KUA must undergo a full IS Audit by a CERT-IN empanelled auditor covering all requirements in Article 04 of this Knowledge Base. The auditor produces a Certificate of Compliance and an Audit Report. Both documents must be submitted to UIDAI — no live transactions are permitted until UIDAI acknowledges receipt and approves. The audit timeline is typically 4–8 weeks from engagement to certificate issuance.
See Article 04 — CERT-IN Audit Checklist for full details
4
Unique Sub-KUA Code — issued by UIDAI, embedded in every transaction
Upon approval, UIDAI issues a unique Sub-KUA code. Per § Circular 1 of 2026, this code must be embedded in the sa attribute of every eKYC request XML. This enables UIDAI to track, at transaction level, exactly which Sub-KUA initiated each request — creating a complete audit trail from resident consent to CIDR and back. Any transaction without a valid Sub-KUA code in the sa field will be rejected by the staging and production environments.
